# 百度 API 主通道与浏览器降级设计

## 背景

当前小时报和日报通过 Chrome 自动读取百度推广数据，功能成熟但速度较慢，并依赖浏览器登录态和页面结构。服务商应用 `openBD` 已审核生效，昆明牛已完成 OAuth 授权、API 只读探测和同日数据对账：三个账户的展现、点击、消费与浏览器抓数完全一致。

本次升级把百度 API 建设为第一优先级数据源，同时完整保留现有浏览器抓数作为 API 不可用时的自动降级通道。Excel 合并、备份、写入和复核边界保持不变。

## 目标

1. 项目显式启用后，小时报和日报优先通过百度 API 读取数据。
2. 在降级浏览器前完成有界、可解释的自动修复。
3. API 与浏览器都失败时停止流程，不写 Excel。
4. 安全完成九个项目、十一个超管的授权、识别和数据对账。
5. 沈阳白、沈阳牛的四个来源分别授权、分别读取、分别校验，最后按现有项目规则合并。
6. 为后续多项目并行保留线程安全的令牌存储和独立任务边界，但本阶段不开发并行执行。

## 非目标

1. 本阶段不移除或弱化浏览器抓数能力。
2. 本阶段不修改快商通人工导出流程。
3. 本阶段不修改 Excel 区域识别、备份、写入和复核规则。
4. 本阶段不自动分析投放效果或生成调整建议。
5. 本阶段不同时运行两个以上项目。
6. 本阶段不把百度服务商应用的 secretKey 分发到桌面客户端。

## 总体架构

采用薄适配层方案，在现有 `run_half_auto_pipeline` 和 `run_daily_pipeline` 的 `fetch_baidu_func` 注入边界前增加统一的百度数据源路由器。路由器只负责选择和执行百度数据读取，不进入快商通解析、数据合并或 Excel 写入模块。

```text
项目运行配置
  -> 百度数据源路由器
      -> API Token 管理器
          -> SCF Token 刷新接口
      -> 百度报表 API 读取器
      -> API 完整性校验
      -> 必要时浏览器读取器
  -> 现有百度标准报告
  -> 现有快商通解析
  -> 现有数据合并
  -> 现有 Excel 备份、写入与复核
```

API 与浏览器必须输出相同的标准账户数据结构，使 pipeline 后半段无需感知数据来源。

## 项目模式

每个项目通过显式配置选择百度数据模式。导入 Token 本身不得改变项目运行模式。

| 模式 | 行为 | 用途 |
| --- | --- | --- |
| `browser` | 始终使用现有浏览器抓数 | 未授权项目和紧急人工回退 |
| `api_shadow` | API 与浏览器均读取，以浏览器结果继续流程并生成对账报告 | 新项目灰度验收 |
| `api_preferred` | API 优先；有界自修复失败后自动降级浏览器 | 验收通过后的正式模式 |

缺少模式或模式值无效时按 `browser` 处理，并输出配置告警。任何项目从 `browser` 切换到 `api_preferred` 前，必须先完成 `api_shadow` 对账。

## Token 生命周期与 SCF 刷新

昆明牛实测 accessToken 有效期为 24 小时，refreshToken 有效期为 30 天。桌面端必须在 accessToken 到期前自动刷新，不能依赖每天重新授权。

百度服务商应用 secretKey 只保存在腾讯云 SCF 环境变量中。SCF 增加固定刷新路径：

```text
POST /baidu/oauth/refresh
```

桌面端向刷新接口提交当前授权所需的最小字段。SCF 校验请求后，使用服务商 appId、secretKey 和 refreshToken 调用百度令牌接口，只返回新的令牌及有效期字段。SCF、API 网关和桌面日志均不得记录完整请求、Token 或 secretKey。

刷新接口使用独立的可轮换客户端密钥进行请求认证。该密钥可以随公司内部授权配置包分发，但不得等同于百度服务商 secretKey。所有传输必须使用 HTTPS。

本地 Token 管理器执行以下流程：

1. 读取当前 API profile，不在内存中缓存跨任务的 secrets 快照。
2. accessToken 距到期不足安全窗口时主动刷新；安全窗口默认 10 分钟。
3. 刷新成功后获取文件锁，重新读取最新 secrets，只合并当前 profile。
4. 先生成备份，再写临时文件并通过 `os.replace` 原子替换。
5. 新 refreshToken 存在时必须同步替换，避免继续使用已轮换的旧令牌。
6. 刷新失败不得破坏原 secrets；后续由数据源路由器决定降级浏览器。

文件锁必须支持未来多个项目进程同时刷新不同 profile，避免后写入者覆盖先写入者的更新。

## API 数据读取与标准输出

现有 `fetch_baidu_api_probe` 保留为只读诊断入口。生产模式新增小时报和日报适配器：

```text
fetch_baidu_api_hourly(config, root, logger, period)
fetch_baidu_api_daily(config, root, logger, target_date)
```

适配器复用 API 请求、账户映射和汇总校验逻辑，并分别输出 pipeline 当前使用的标准文件：

```text
reports/baidu_account_data.json
reports/baidu_daily_data.json
```

标准报告增加但不依赖以下诊断字段：

- `data_source`: `api`、`api_shadow` 或 `browser_fallback`。
- `api_profile`: 本次使用的 profile，不包含凭据。
- `attempts`: API 请求次数。
- `self_heal_actions`: 刷新、重试等动作的分类名称。
- `fallback_reason`: 降级原因的安全摘要。
- `token_refresh`: `not_needed`、`refreshed`、`failed` 或 `reauthorization_required`。

不得把 accessToken、refreshToken、客户端密钥、百度密码或完整 API 请求头写入任何报告和日志。

## 完整性校验

API 返回满足以下条件才可进入合并与 Excel 写入：

1. API 状态成功且没有 failures。
2. 返回日期与请求日期一致。
3. 返回账户推广 ID 集合与当前 API profile 所属账户集合完全一致。
4. 不允许未知推广 ID、重复账户或缺失账户。
5. 展现、点击为非负整数，消费为有限非负数。
6. API 汇总展现、点击、消费与账户求和一致；消费容差不超过 0.01。
7. 全部账户为零可以视为合法数据，但账户行和汇总结构必须完整。
8. 双来源项目的每个来源必须单独通过上述校验，任一来源失败时不得使用部分 API 数据。

API 失败产生的临时报告不得覆盖上一次成功的生产标准报告。只有当前数据源最终成功后，才原子写入标准报告路径。

## 有界自修复与降级

错误按类别处理，所有重试都必须有明确上限。

| 错误类别 | 自修复动作 | 最终动作 |
| --- | --- | --- |
| accessToken 即将到期或已到期 | 通过 SCF 刷新一次，再重试 API | 失败则降级浏览器 |
| 网络超时、连接中断、HTTP 5xx | 短退避后最多重试两次 | 仍失败则降级浏览器 |
| 账户缺失、汇总不一致、响应暂不完整 | 短暂等待后重新读取一次 | 仍异常则降级浏览器 |
| refreshToken 失效、授权撤销、权限不足 | 不重复刷新，标记需要重新授权 | 立即降级浏览器 |
| API profile 缺失或配置不合法 | 记录配置错误 | 降级浏览器 |
| 主体或推广 ID 映射冲突 | 禁止接受 API 数据 | 降级浏览器并标记高优先级告警 |

一次任务中 API 自修复总耗时预算固定为 20 秒，避免 API 故障反而让任务比原浏览器模式更慢。超过预算立即停止 API 重试并进入浏览器降级。

浏览器降级沿用现有 `fetch_baidu_auto` 和 `fetch_baidu_daily`。降级成功后可以继续合并与写入，但 GUI、日志和最终报告必须明确显示“API 不可用，已使用浏览器完成”。浏览器也失败时立即停止，后续快商通合并和 Excel 写入不得执行。

## 影子对账

`api_shadow` 同时读取 API 和浏览器，但只使用浏览器结果继续现有流程。对账报告按账户比较：

- 推广 ID。
- 展现。
- 点击。
- 消费，容差 0.01。
- 汇总值。

任一项不一致时，项目不得切换到 `api_preferred`。日报至少完成一个完整日对账；小时报至少覆盖 11 点、15 点、18 点三个时段。对账报告不得包含凭据。

## 十一个超管授权防错

剩余授权全部使用已生效的服务商应用 `openBD`。授权文件导入不再依赖操作者手工输入 profile，而是优先自动识别：

1. 扫描九个正式项目配置，构建每个 API profile 的预期推广 ID 集合。
2. 单来源项目取项目账户集合；双来源项目按每个 `baidu_sources` 来源分别取集合。
3. 使用授权文件中的 appId、超管 userId、主体名称和子账户推广 ID 集合进行匹配。
4. appId 必须是当前允许的服务商应用。
5. 只有唯一 profile 完全匹配时才允许导入。
6. 无匹配或多个匹配时停止并输出不含 Token 的诊断，不允许猜测。
7. 导入前备份 secrets，导入后输出 profile、主体、子账户数量和推广 ID 校验结果。
8. 下载的 `.baidu-auth` 文件导入成功后提示立即删除。

授权及灰度顺序固定为：昆明牛、长沙牛、南京牛、宁波牛、青岛白、南京白、深圳白、沈阳白来源 A、沈阳白来源 B、沈阳牛中亚、沈阳牛银康。

## 双来源项目

沈阳白和沈阳牛不能把两个超管的 Token、账户或结果混为一个来源。API 路由器对每个来源独立执行：

```text
来源 A Token -> 来源 A 推广 ID -> 来源 A 校验结果
来源 B Token -> 来源 B 推广 ID -> 来源 B 校验结果
```

两个来源都成功后，按标准账户名合并为一个项目百度报告。账户名或推广 ID 冲突必须停止。API 任一来源失败时，第一阶段采用“整个项目降级浏览器”，不得把一个来源的 API 数据与另一个来源的浏览器数据混合，以降低组合路径数量和验收风险。

## GUI 与日志

GUI 当前流程区展示以下状态：

- 正在检查 API 授权。
- 正在刷新百度授权。
- 正在通过 API 读取数据。
- API 数据校验通过。
- API 暂不可用，正在切换浏览器。
- 浏览器降级成功。
- 百度 API 授权需要更新。

宠物进度文本可复用上述安全摘要，不得显示 URL 查询参数、Token、账号密码或服务商密钥。

最终报告必须记录实际数据来源和降级原因，使同一份 Excel 数据可以追溯是 API 还是浏览器产生。

## 测试策略

实现采用 TDD，至少覆盖：

1. 有效 Token 不刷新并成功调用 API。
2. 临近到期 Token 刷新后重试成功。
3. 刷新返回新 refreshToken 时同时原子更新两个令牌。
4. 刷新失败不修改 secrets。
5. 多进程或竞争写入时只更新目标 profile，不丢失其他 profile。
6. API 网络错误按上限重试，随后浏览器降级。
7. API 汇总不一致重读一次，仍失败后浏览器降级。
8. API 成功时不启动 Chrome。
9. API 和浏览器都失败时 pipeline 停止且不调用写 Excel。
10. `api_shadow` 只生成对账报告，不改变正式数据来源。
11. 全零账户结构完整时通过校验。
12. 授权文件唯一匹配、无匹配、多匹配和错误 appId。
13. 沈阳双来源全部成功、单来源失败、账户冲突和整项目浏览器降级。
14. 报告与日志不包含 Token、密码、客户端密钥或 secretKey。
15. 现有浏览器模式回归测试保持通过。
16. 现有小时报、日报、Excel 备份与写入测试保持通过。

昆明牛真实灰度验收不得直接写 Excel。先执行只读 API 探测和模拟写入预览，再运行 `api_shadow`。用户明确确认对账结果后，才允许切换 `api_preferred` 并执行真实小时报或日报。

## 实施阶段

### 阶段一：令牌和单项目基础设施

1. 实现 SCF 刷新接口及安全测试。
2. 实现本地 Token 管理器、文件锁、备份和原子更新。
3. 实现 API 小时与日报标准输出适配器。
4. 实现数据源路由器、自修复、浏览器降级和诊断字段。

### 阶段二：昆明牛灰度

1. 昆明牛配置为 `api_shadow`。
2. 完成日报完整日和三个小时段对账。
3. 对账通过后人工确认切换 `api_preferred`。
4. 验证 API 失败时浏览器自动降级。

### 阶段三：剩余授权与九项目灰度

1. 实现授权文件自动 profile 匹配。
2. 依固定顺序完成剩余十个超管授权。
3. 每个项目分别完成 API 探测、影子对账和显式启用。
4. 最后验收沈阳白、沈阳牛双来源合并。

### 阶段四：多项目执行

只有九项目单项目模式稳定后再单独设计。后续多项目方案必须复用本设计的独立运行上下文、Token 文件锁、数据源路由器和浏览器降级能力，不得在本阶段提前加入线程或浏览器实例调度。

## 发布与回退

1. 本功能通过新的在线更新版本发布，不覆盖用户项目配置和 secrets。
2. 默认项目模式保持 `browser`，因此升级程序本身不会改变现有同事的抓数行为。
3. 授权配置继续通过独立 `.baidu-secrets` 包在公司内部导入导出。
4. 任一项目出现异常时，可把模式改回 `browser`，无需卸载版本或删除 Token。
5. 现有 HERMES BAT、GUI 入口和浏览器抓数调用路径保持兼容。

## 验收标准

1. 昆明牛在有效 Token 下不启动 Chrome，并完成 API 只读抓数。
2. Token 到期前可以自动刷新，无需人工每日授权。
3. API 故障时自动执行有界自修复，随后浏览器降级成功。
4. API 与浏览器都失败时不写 Excel。
5. 九项目十一超管均能唯一映射到正确 API profile。
6. 沈阳四个来源分别授权、分别读取，项目合并结果与浏览器一致。
7. 浏览器模式全部既有测试通过。
8. 日志、报告和发布包不泄露任何敏感凭据。
9. 未显式启用 API 的项目行为与当前稳定版本一致。
