# 全局 API 模式、双来源回退与 GUI 流式状态设计

## 设计关系

本文是 `2026-07-17-baidu-api-primary-fallback-design.md` 的增量设计。原设计中的 OAuth、Token 刷新、API 完整性校验、20 秒自修复预算、浏览器降级、敏感信息保护和 Excel 写入边界继续有效。

本文覆盖原设计中以下已过时约定：

1. 普通用户不再逐项目编辑 `data_source_mode`。
2. 九个正式项目、十一个超管完成授权后，由应用级 `A/B` 开关统一决定生产任务的数据源偏好。
3. 缺少全局配置项时默认采用 `A`，但 API 授权或项目映射不完整时必须安全降级浏览器。

## 目标

1. 用户可见产品名称正式改为“蚁之力 · 竞价数据自动化”。
2. GUI 将“小时报选择”简化为“小时报”，将“日报执行”简化为“日报”。
3. 系统配置提供全局 `B/A` 滑动分段开关，统一控制 GUI、HERMES BAT 和命令行完整任务。
4. `A` 默认 API 优先，失败时有界自修复并自动降级浏览器。
5. `B` 强制浏览器，不发起 API 请求。
6. 沈阳牛、沈阳白分别读取两个 API 来源，全部成功才合并；任一来源失败时整项目降级浏览器。
7. GUI 实时日志按行流式输出，明确显示 API、自修复、降级和浏览器阶段。
8. 保留现有 EXE 文件名、安装目录名、GitHub Release tag 和在线更新 ZIP 命名，避免破坏旧更新链路。

## 非目标

1. 本阶段不开发多项目并行执行。
2. 本阶段不并发读取沈阳双来源；先按来源顺序读取并保证合并正确。
3. 本阶段不改快商通人工导出方式。
4. 本阶段不改 Excel 扫描、备份、写入和复核边界。
5. 本阶段不重命名 `百度数据自动化控制台.exe` 或现有发布资产。
6. 本阶段不移除 `api_shadow` 开发对账能力。

## 方案选择

采用应用级持久化模式，不在每次切换时批量重写九份项目 JSON，也不只通过 GUI 临时启动参数控制。

`configs/app_config.json` 新增：

```json
{
  "baidu_data_source_preference": "api"
}
```

允许值只有 `api` 和 `browser`。缺失时按 `api`，非法值按 `api` 并记录配置告警。选择变化使用现有 JSON 写入方式保存，保留其他应用配置字段。

生产完整任务的有效模式如下：

| 全局选择 | 有效行为 |
| --- | --- |
| `A` / `api` | API 优先；配置、授权、Token、网络或完整性失败时整项目降级浏览器 |
| `B` / `browser` | 强制浏览器；不调用 API |

项目 JSON 中的 `api_profile` 和双来源 `api_profile` 继续表示授权映射。原 `data_source_mode` 暂时保留用于兼容测试和显式开发入口，但普通生产 pipeline 以全局偏好为准。`api_shadow` 只供显式开发命令使用，不在 GUI 暴露。

## 配置与入口一致性

全局偏好必须在项目运行配置加载阶段合并，不能只由 GUI 传环境变量。以下入口必须得到相同结果：

- GUI 小时报。
- GUI 日报。
- `run_hermes_hourly.bat`。
- `run_hermes_daily.bat`。
- `main.py --mode run`。
- `main.py --mode run-daily`。

显式诊断入口 `test-baidu-api`、`simulate-baidu-api-hourly` 和 OAuth 导入不受全局开关影响。

## 单来源 API 路由

`A` 模式下单来源项目执行：

1. 校验项目存在唯一 `api_profile`。
2. 读取 Token，按安全窗口决定是否刷新。
3. 调用百度 API 并校验日期、推广 ID、账户完整性、数值类型和汇总一致性。
4. 成功后原子提交标准百度报告，浏览器不得启动。
5. 失败时按错误分类执行有界自修复。
6. 超过 20 秒总预算或确认不可恢复后，记录安全原因并调用现有浏览器读取器。
7. 浏览器也失败时停止 pipeline，不进入快商通合并和 Excel 写入。

缺少授权不能让任务直接崩溃。`A` 表示“API 优先且可自动回退”，不是“只能使用 API”。

## 双来源原子路由

沈阳牛和沈阳白使用同一套原子流程：

1. `build_source_runtime_config` 必须把来源自己的 `credential_profile`、`api_profile`、账户集合和来源专属临时输出路径注入运行配置。
2. 两个来源依次调用 API，每个来源独立校验推广 ID、账户、汇总和错误。
3. API 临时结果写入来源专属临时报告，不得提前覆盖统一标准报告。
4. 两个来源全部通过后，复用现有 `aggregate_baidu_source_reports` 合并。
5. 合并后再次校验账户唯一性、必需账户集合和来源消费合计。
6. 全部校验通过后，才原子提交统一标准报告。
7. 任一来源失败时，丢弃本次所有 API 临时结果，并对整个项目调用现有双来源浏览器流程。
8. 禁止一个来源使用 API、另一个来源使用浏览器后混合写入。

20 秒自修复预算按整个项目计算，不为每个来源重新计时。双来源首版顺序执行，避免在本阶段引入并发和共享报告竞争。

## 延迟启动 Chrome

`A` 模式的快速预检只检查运行环境、项目配置、Excel、快商通目录、secrets JSON 和所需 API profile，不提前连接或启动 Chrome。

只有 API 最终失败、路由器决定降级时，才执行现有 Chrome 9222 就绪逻辑。这样正常 API 任务不再支付浏览器启动和登录等待成本。

`B` 模式继续执行现有 Chrome 快速预检。完整预检仍可显式检查 Excel 结构。

## GUI 设计

用户可见名称改为“蚁之力 · 竞价数据自动化”，覆盖：

- 主窗口标题和标题栏。
- QApplication 名称。
- 托盘提示。
- 桌面宠物提示。
- 启动错误对话框标题。
- 用户说明文档中的产品称呼。

以下兼容名称不变：

- `百度数据自动化控制台.exe`。
- 现有根目录和安装目录名。
- GitHub Release tag 规则。
- `baidu_data_automation_update_<版本号>.zip`。

系统配置菜单顶部增加一行紧凑的 `B / A` 分段滑动开关：

- 左侧 `B` 表示浏览器。
- 右侧 `A` 表示 API。
- 选中项由移动滑块和颜色共同表达。
- 默认选中 `A`。
- 切换后立即持久化，但只影响下一次任务。
- 任务运行期间禁用开关，防止改变正在执行任务的通道。
- 鼠标提示分别显示“强制浏览器”和“API 优先，失败自动切换浏览器”。

左侧板块标题改为：

- `小时报选择` -> `小时报`。
- `日报执行` -> `日报`。

## 流式日志与进度事件

GUI 子进程环境增加：

```text
PYTHONUNBUFFERED=1
PYTHONUTF8=1
PYTHONIOENCODING=utf-8
```

Python 启动入口在适用时增加 `-u`。`QtTaskRunner` 必须缓存未结束的行，只在收到换行或进程结束时发送完整文本，避免 QProcess 数据块把一行拆成多条日志。

业务日志使用稳定、安全的中文阶段文本：

```text
[数据源] 当前模式：API 优先
[API] 正在读取昆明牛百度数据
[API] Token 已刷新，正在重新请求
[API] 已读取 3 个账户，完整性校验通过
[降级] API 读取仍未完成，准备切换浏览器
[浏览器] 正在连接 Chrome 9222
[浏览器] 正在切换百度账号
[通过] 百度数据读取完成，实际来源：API
```

双来源增加来源序号：

```text
[API 1/2] 正在读取沈阳牛中亚来源
[API 2/2] 正在读取沈阳牛银康来源
[API] 两个来源已合并，完整性校验通过
```

日志与宠物事件不得包含 Token、密钥、密码、完整授权 URL 或请求头。API 成功时不显示“正在切换百度账号”；只有浏览器降级后才显示登录阶段。

## 数据报告与追溯

标准百度报告继续记录：

- `data_source`：`api`、`browser`、`browser_fallback` 或 `failed`。
- `api_attempts`。
- `self_heal_actions`。
- `fallback_reason` 的安全摘要。
- 双来源的来源数量和来源校验状态。

API 失败产生的临时结果不得覆盖上一次成功的标准报告。最终报告必须能确认本次 Excel 数据来自 API 还是浏览器降级。

## 测试与验收

实现采用 TDD，至少覆盖：

1. 缺少全局配置时默认 `api`。
2. GUI 切换后保存，重启后保持选择。
3. `B` 强制浏览器且 API 调用次数为零。
4. `A` 单来源 API 成功时浏览器调用次数为零。
5. `A` 缺少授权时降级浏览器。
6. API Token、网络和完整性错误按既定上限自修复。
7. API 与浏览器均失败时 pipeline 不调用快商通合并或 Excel 写入。
8. 双来源全部成功后正确合并并原子提交。
9. 双来源任一失败后丢弃 API 结果，整项目调用浏览器。
10. 双来源账户冲突或消费合计异常时禁止接受 API 数据。
11. `A` 模式快速预检不启动 Chrome。
12. `B` 模式保留现有 Chrome 预检。
13. QProcess 分块输出不会产生半行或丢失末行。
14. GUI 新产品名、板块标题和 A/B 开关状态正确。
15. 日志与报告不包含敏感凭据。
16. 现有浏览器、日报、小时报、Excel 备份和写入测试继续通过。

真实授权验收顺序：

1. 对十一个 API profile 逐个执行只读 Token/API 探测。
2. 对七个单来源项目校验账户集合和汇总。
3. 对沈阳牛、沈阳白分别校验两个来源及项目合并结果。
4. 人为注入 API 失败，验证浏览器整项目降级。
5. 不执行真实 Excel 写入，除非用户另行明确要求。
6. 全量运行 `tests/test_basic.py`。
7. 构建 EXE 供用户检查 GUI；用户确认前不生成内部发布包。

## 回退规则

出现生产异常时，用户可在系统配置中切换到 `B`，下一次任务立即恢复现有浏览器流程，不需要删除 OAuth Token、修改九份项目配置或回退程序版本。

在线更新兼容名称保持不变，因此本功能不处理更新器重命名；该事项留到后续独立优化。
